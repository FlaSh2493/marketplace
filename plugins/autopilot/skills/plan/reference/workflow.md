# Deep Plan Workflow

## 1. Context Gathering

탐색 순서를 지켜라:
1. **이슈 로드**: `python3 ${CLAUDE_PLUGIN_ROOT}/skills/plan/scripts/load_issue.py {issue_key} --sections 배경,목표,비목표,요구사항,인수 조건,참고,제약/고려사항`
2. **Entry points**: README, 메인 진입점, 라우팅 확인
3. **유사 구현**: 기존 코드베이스에서 유사 기능 검색 (`semantic_search` 등 활용)
4. **인터페이스**: 변경이 닿을 API, 타입, 스키마 정의 확인
5. **테스트**: 관련 기존 테스트 코드 확인
6. **설정**: 관련 env, feature flag, 환경 설정 확인

규칙:
- Glob → 좁히기 → Read 순서. 파일을 한 번에 다 읽지 마라.
- 탐색은 좁히고, 깊이 읽지 않는다. 시그니처·타입만 파악하면 본문은 스킵.
- 5-10개 파일에서 멈추고 종합. 무한정 열지 마라.
- `node_modules`, `dist`, `build` 등 빌드 결과물은 제외하라.
- 탐색하면서 각 변경 영역의 파일 수를 카운트해둔다 (Phase의 `scope` 추정 근거).

## 2. Clarification

`AskUserQuestion`으로 핵심만 묻는다. 최대 3개. 
- Multiple-choice 형태로 옵션 제공 권장.
- 답을 받기 전 다음 단계로 가지 않는다.
- 요구사항이 명확하면 스킵 가능.

## 3. Design Analysis

**최소 2개 대안을 비교한다.**
각 대안에 대해:
- 한 줄 요약
- 영향 범위 (파일 수, 변경 성격)
- 트레이드오프 (리스크/복잡도/성능)

분석 결과는 Plan 문서의 `## 접근 방식`에 기록한다 (선택 + 기각 이유).

## 4. Phase 설계

### 분할 규칙
- `scope: large` + `output_shape: detail`이면 분할한다 (가장 명확한 신호).
- 레이어가 교차하거나 선행 의존성이 있는 경우 Phase로 격리한다.
- 각 Phase는 독립적으로 검증 가능해야 한다.
- 그 외 변경량이 많거나 컨텍스트가 무거우면 분할을 검토한다.

### Phase 메타 (각 Phase에 필수)

**depends_on** — 다른 Phase의 코드를 직접 참조(호출·import·같은 파일 수정)하는가:
- `[]` — 독립. 병렬 실행 가능.
- `[Phase N, ...]` — 해당 Phase 완료 후 실행.
- 애매하면 의존 있다고 적는다. 잘못 위임해서 디테일을 잃는 게 위임 안 하는 것보다 손해가 크다.

**scope** — 구현 시 읽거나 건드려야 할 파일의 규모 (처음 보는 사람 기준으로 추정):
- `small` — 1~3 파일
- `medium` — 4~10 파일
- `large` — 10 파일 초과 또는 큰 단일 파일

**output_shape** — 다음 Phase가 이 결과를 어떻게 쓰는가:
- `interface` — 시그니처·타입 몇 개로 요약 가능
- `detail` — 디테일이 살아있어야 다음 Phase가 작동

### 서브에이전트 분기 결정
Plan은 메타만 정확히 적는다. 분기는 구현 하네스가 아래 규칙으로 자동 결정:

세 조건이 **모두** 충족되면 → **서브에이전트 위임**:
1. `depends_on`이 `[]` (독립)
2. `scope`가 `medium` 이상
3. `output_shape`가 `interface`

그 외 → **메인 처리**. 하나라도 어긋나면 메인.

### Phase 의존성 그래프
`## Phase 의존성` 섹션에 ASCII 화살표로 시각화한다. 형식 고정:
```
Phase 1 ──► Phase 3
Phase 2 ──► Phase 3
```

## 5. Plan 문서화

`../templates/template.md` 형식으로 `EnterPlanMode`를 통해 작성한다.

작성 시 주의:
- 각 단계의 "변경"은 `Before → After` 형태로 적는다. "한다"가 아니라 "무엇이 어떻게 변한다".
- 각 단계의 "검증"은 빌드·타입체크 수준의 빠른 확인. 신규 테스트 계획은 `## 테스트`에.
- `## 배경`은 이슈 로드 결과에서 1-2문장 압축.
- `## 현재 상태`는 탐색한 5-10개 파일을 3-5문장으로 압축.

### Build 제약 (Plan에 명시)
- 출력 형식: 코드만 출력. 설명 금지. 변경 파일은 전체 내용을 출력한다.
- 범위 이탈 금지: 계획에 없는 리팩토링·import 정리·시그니처 변경·주석 추가를 하지 않는다.

## 6. Self-Check

다음을 모두 YES로 답할 수 있어야 한다:

**Phase 메타**
- [ ] 모든 Phase에 `depends_on`, `scope`, `output_shape`가 명시됐다
- [ ] Phase 의존성 그래프가 있다
- [ ] `scope: large` + `output_shape: detail` Phase가 없다 (있으면 분할)

**기본**
- [ ] 변경할 파일이 모두 명시됐다 (생성/수정/삭제)
- [ ] 각 단계가 "뭘 한다"가 아니라 "뭐가 어떻게 변한다"로 적혔다
- [ ] 모호한 단어("개선", "정리", "최적화") 없이 구체적이다

**검증·리스크**
- [ ] 각 단계에 검증 방법(어떻게 성공을 확인할지)이 있다
- [ ] 대안 1개 이상을 검토하고 기각 이유를 적었다
- [ ] Out of scope를 명시했다
- [ ] 가장 큰 리스크 2-3개와 mitigation을 적었다

## 7. Plan 저장

1. `EnterPlanMode` 호출하여 작성.
2. `ExitPlanMode` 승인 후:
   - 저장 경로: `tasks/{issue_key}/plan.md`
   - 만약 `{issue_key}`가 없으면 사용자 요구사항의 핵심 키워드를 사용한다.
