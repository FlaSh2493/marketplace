# merge-strategy

**목적**: git 병합 충돌 발생 시 해결 전략을 선택하는 기준을 정의한다.

## 전략 종류

| 전략 | 설명 | 적용 명령 |
|---|---|---|
| `ours` | 현재 브랜치(대상) 코드 유지 | `git checkout --ours {file}` |
| `theirs` | 들어오는 브랜치(feature) 코드 적용 | `git checkout --theirs {file}` |
| `manual` | 두 변경을 통합 (LLM 제안 후 사용자 승인) | str_replace로 직접 수정 |
| `user` | 사용자가 직접 해결 | 편집기로 열기 권장 |

## 전략 선택 기준

### `ours` 선택
- 들어오는 변경이 단순 리팩토링이고 대상 브랜치가 핵심 비즈니스 로직 포함
- 대상 브랜치가 이미 배포된 코드 (hotfix 상황)

### `theirs` 선택
- 들어오는 변경이 버그 수정이고 대상 브랜치가 구버전
- feature 브랜치가 더 최신 API 사용

### `manual` (LLM 통합) 선택
- 두 변경이 모두 유효한 새 기능
- 충돌 블록이 5줄 이내이고 의미가 명확한 경우
- 로직을 합치는 것이 가능한 경우

### `user` 선택
- 비즈니스 로직 판단이 필요한 경우
- 충돌 블록이 20줄 초과
- 두 버전의 의도를 파악하기 어려운 경우

## 프로세스

1. 충돌 블록만 추출 (전체 파일 X)
2. 변경 의도 파악 (커밋 메시지, 파일명 기준)
3. 위 기준으로 전략 선택
4. 사용자에게 제안 + 이유 설명
5. 사용자 승인 후 적용
6. 적용 후 반드시 빌드 검증
