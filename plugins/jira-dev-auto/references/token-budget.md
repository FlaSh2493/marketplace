# token-budget

**목적**: 단계별 토큰 예산을 정의하고 CLI 선행 처리로 토큰을 절약하는 원칙을 기술한다.

## 4대 절약 원칙

### 1. CLI 선행 처리
LLM에 넘기기 전 반드시 CLI로 추출.
```
파일 전체 (~2000 tok) → 함수 시그니처만 (~200 tok)  → 90% 절약
프로젝트 소스 (~50k tok) → tree + exports 요약 (~500 tok) → 99% 절약
```

### 2. MCP 결과 캐싱
- 모든 MCP 조회 결과를 `_cache/` JSON에 저장
- 해시 비교로 변경 시에만 LLM 재처리
- 같은 세션 내 동일 티켓 재조회 금지

### 3. Skill 지연 로드
- 해당 단계에서만 필요한 Skill을 로드
- 동시에 최대 2개까지만
- 각 Skill/Reference 파일은 200줄 이내 유지

### 4. 서브에이전트 최소 컨텍스트
서브에이전트에 전달하는 것:
- `requirement.yaml` 1개 (해당 티켓만)
- `plan.yaml`의 해당 task 섹션만
- reference 파일 1-2개

절대 전달하지 않는 것:
- 다른 티켓의 requirement
- 전체 프로젝트 소스
- 오케스트레이터 대화 히스토리

## 단계별 토큰 예산

| Phase | 예산 | 비고 |
|---|---|---|
| 1. 수집 | ~2,500 | MCP 결과 요약만 |
| 2. 분석 | ~4,000 | tree 요약 + 티켓 summary만 |
| 3. 정제 (티켓당) | ~3,500 | description 전문 허용 |
| 4. 계획 | ~5,000 | requirement 키 목록만 |
| 5. 구현 (티켓당) | ~12,000 | 최대 허용치 |
| 6. 변경 감지 | 0~3,000 | halt 시에만 LLM |
| 7. 병합 (충돌파일당) | ~1,500 | 블록만 입력 |
| 8. 조회 | 0 | CLI만 |

## 예산 초과 시 조치
- 코드 생성 중 예산 접근 시: 현재 파일 완료 후 커밋, 다음 파일로 분할
- 분석 중 예산 접근 시: 요약 출력 후 청크 단위로 분할
