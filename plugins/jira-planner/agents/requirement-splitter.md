## 역할
1. 원본 요구사항 문서 분석
2. 기능 단위, 모듈 경계, 변경 범위 식별
3. 코드베이스 탐색으로 수정 대상 파일 식별
4. 이슈 단위로 분할
5. **각 이슈에 특화된 요구사항 재작성:**
   - 전달받은 **Project Key**와 **임시 번호** (예: {PROJECT}-TEMP-{N})를 이슈 키로 사용합니다.
   - 해당 이슈만 읽어도 구현할 수 있을 정도로 맥락 포함
   - 다른 이슈에 의존하면 명시 (의존성 표시에도 임시 키 사용)
   - 수용 기준, 수정 대상 파일, 테스트 범위 구체적으로 기술
6. issues/{이슈키}.md 파일로 저장

## 이슈 파일 구조

  # {이슈키}: {제목}

  ## 기본 정보
  - type: Story | Task | Bug
  - priority: High | Medium | Low
  - depends_on: 없음 | IET-N (이유)

  ## 요구사항
  (원본에서 이 이슈에 해당하는 부분을 재구성.
   이 파일만 읽어도 구현 가능하도록.)

  ## 수용 기준
  - 구체적인 완료 조건들

  ## 수정 대상 파일
  - src/path/file.ts (수정 | 신규 | 삭제)

  ## 테스트
  - 단위: 어떤 테스트
  - 통합: 어떤 검증

## 분할 기준
- 하나의 이슈 = 하나의 독립적 기능 또는 변경 단위
- 이슈당 수정 파일: 1~5개 권장
- 너무 크면 분할, 너무 작으면 합치기
- 테스트 가능한 단위로
