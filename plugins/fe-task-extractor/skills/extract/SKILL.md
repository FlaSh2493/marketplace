# Frontend Task Extractor

기획서·PRD·요구사항 문서에서 **프론트엔드 작업만 식별·추출**하여 **요구사항 명세**를 생성한다. 구현 계획·AC는 다루지 않는다 — plan mode에서 처리.

출력:
1. 이슈별 개별 마크다운 파일 → `{프로젝트 루트}/.docs/task/{브랜치명}/FE-XX.md`
2. Jira Story → Atlassian MCP (생성 후 `FE-XX.md` → `{JIRA-KEY}.md`로 리네임)

---

## 🚨 절대 준수 사항 (Strict Adherence)

이 스킬은 단순히 작업을 수행하는 것이 아니라, **정해진 표준화 절차를 100% 준수**해야 한다. 아래 사항을 어길 경우 후속 플러그인(`worktree-flow`)에서 데이터 로딩에 실패할 수 있다.

1. **템플릿 우선 (Template First)**: 파일 생성/수정 전 **반드시** `templates/fe-task-template.md`를 `view_file`로 읽고, 헤더 필드 순서(`jira`, `상태`, `담당자` 등 6개)와 구분선(`---`) 형식을 완벽히 일치시킨다.
2. **스크립트 필수 실행 (Must Use Scripts)**: 
   - 경로 생성 시 `mkdir` 대신 `${CLAUDE_PLUGIN_ROOT}/scripts/init_task_dir.py`를 찾아 실행한다.
   - Jira 생성 후 리네임 시 반드시 `${CLAUDE_PLUGIN_ROOT}/scripts/update_jira_keys.py`를 실행하여 파일명과 내용을 동시 업데이트한다.
3. **세분화 규칙**: 작업을 너무 크게 뽑지 않는다. 서로 다른 페이지, 독립된 기능은 분리한다.
4. **명시적 승인 (Gate 1)**: Jira 티켓 생성 전 반드시 **사용자의 명시적 승인**을 테이블 형식의 보고 후 받는다.

---

## 🛠️ 경로 관리 가이드 (중요)

이 플러그인의 스크립트는 **플러그인 설치 폴더 내부의 `scripts/` 디렉토리**에 위치합니다.
에이전트는 이 `SKILL.md` 파일의 위치를 기준으로 상위 폴더들을 탐색하여 `scripts/` 폴더 내의 스크립스(`init_task_dir.py`, `update_jira_keys.py` 등)를 찾아야 합니다.

커맨드 실행 시, **스크립트의 절대 경로**를 확보하여 실행하십시오. (예: `${CLAUDE_PLUGIN_ROOT}/scripts/init_task_dir.py`)
현재 작업 디렉토리(CWD) 하위에 `scripts/`나 `plugins/` 폴더가 없을 수도 있으므로, 반드시 `${CLAUDE_PLUGIN_ROOT}/scripts/` 경로를 기반으로 스크립트를 찾아야 합니다.

## 1. 입력 처리

**텍스트** → 그대로 분석. **파일** → 파일 읽기 스킬로 추출 후 분석. **노션/컨플루언스 링크** → web_fetch 하지 않고 텍스트 복붙 유도:
> "페이지 내용을 텍스트로 복사해서 붙여넣어 주세요."

## 2. 분석

### Step 1: 전체 이해
기능 목적, 사용자 시나리오, 시스템 구성 파악. 아직 작업을 뽑지 않는다.

### Step 2: 영역 분류 → FE만 추출
- **FE**: UI, 인터랙션, 상태 관리, 클라이언트 API 호출, 라우팅, 폼, 유효성 검사(클라이언트), 반응형/접근성, 에러 UI, 로딩/스켈레톤
- **BE**: API 구현, DB, 비즈니스 로직, 서버 인증/인가
- **디자인**: 와이어프레임, 비주얼, 에셋
- **기획/기타**: 정책, QA, DevOps

### Step 3: 세분화
복잡도에 따라 유동적: 단순 1개 / 중간 2~5개 / 복잡 5~10+. 서로 다른 페이지·컴포넌트 → 분리. 독립 관심사 → 분리 후보. 함께 구현되는 것 → 묶기.

### Step 4: 의존관계·API 매핑
각 작업에 선행/후행 + API 연동 정보 기록.

## 3. 출력

### 3-1. 저장 디렉토리 생성

현재 Git 브랜치명을 가져와서 디렉토리를 생성한다:

```bash
# 현재 브랜치명 확인
git rev-parse --abbrev-ref HEAD

# 플러그인 스크립트로 디렉토리 생성
python3 ${CLAUDE_PLUGIN_ROOT}/scripts/init_task_dir.py "{브랜치명}" [--root {경로}]
```

반환 JSON에서 `dir` 경로를 사용하여 이슈별 파일을 저장한다. 프로젝트 루트를 자동 탐색하며, 못 찾으면 에러를 반환하므로 사용자에게 경로를 물어본다.

### 3-2. 이슈별 개별 마크다운 파일 생성

**반드시** `templates/fe-task-template.md` 파일을 `view_file`로 읽고, 해당 템플릿의 구조·필드 순서·규칙을 **정확히** 따라 마크다운을 생성한다.

```
${CLAUDE_PLUGIN_ROOT}/templates/fe-task-template.md
```

추출된 각 작업을 **개별 md 파일**로 저장한다:

- **파일명**: `FE-{번호}.md` (예: `FE-01.md`) — Jira 미생성 상태
- **저장 경로**: `{프로젝트 루트}/.docs/task/{브랜치명}/FE-{번호}.md`
- Jira 생성 후 `{JIRA-KEY}.md`로 리네임된다 (update_jira_keys.py가 처리)

**extract 전용 필드 매핑:**

| 템플릿 필드 | 소스 |
|------------|------|
| `JIRA-KEY` | `FE-{번호}` (Jira 생성 전 임시 ID) |
| `상태` | `신규` |
| `담당자` | `@본인` |
| `출처` | `extract` (고정) |
| `jira` | `미생성` |

**추가 규칙:**
- 구현 방법, AC, 체크리스트는 **적지 않는다**
- 원문에 없는 작업은 추가하지 않는다

### 3-2-1. 포맷 검증 (저장 전 필수)
각 파일 저장 직전에 `templates/fe-task-template.md`의 **포맷 검증 체크리스트**를 모두 통과하는지 확인한다. 하나라도 불일치하면 수정한 뒤 저장한다.

### 3-3. ✅ Gate 1: 작업 목록 및 명세 확인/승인

마크다운 파일 저장이 완료되면, 추출된 **작업 목록을 아래 표 형식으로 사용자에게 출력**하여 보여준다.

---
### 📋 생성 예정 작업 목록

| ID | 작업 제목 | 주요 설명 | 선행 작업 |
| :--- | :--- | :--- | :--- |
| FE-01 | {제목} | {핵심 요약} | {ID 또는 없음} |
| FE-02 | {제목} | {핵심 요약} | {ID 또는 없음} |

**설정 정보**:
- **Project**: {PROJECT_KEY}
- **Epic**: {목록에서 선택한 에픽 제목 또는 ID}
- **Sprint**: {선택한 스프린트 명칭 또는 "지정 안 함"}
- **담당자**: @본인 (currentUser)
---

사용자에게 다음과 같이 질문하여 명시적인 승인을 받아야 한다:
> "추출된 작업 목록이 .docs/task/{브랜치명}/ 폴더에 개별 파일로 저장되었습니다. 위 목록대로 Jira에 Story 티켓을 생성해도 될까요? 수정을 원하시면 말씀해 주세요."

사용자가 승인(예: "응", "진행해줘", "OK")하기 전에는 **절대 Jira 티켓 생성을 시작하지 않는다.**

- 사용자가 수정을 요청하면 → 마크다운 파일을 수정하거나 작업을 다시 분석한 뒤 다시 보여준다.
- 사용자가 승인하면 → 다음 단계(Jira Story 생성)로 진행한다.

### 3-4. 실행 완료 보고

모든 Jira 티켓 생성이 완료되면 아래 형식으로 최종 보고한다.

### ✅ Jira 작업 등록 완료

| 기존 파일 | Jira Key | 작업 제목 | 담당자 | 리네임 |
| :--- | :--- | :--- | :--- | :--- |
| FE-01.md | [{KEY}](URL) | {제목} | @본인 | → {KEY}.md |
| FE-02.md | [{KEY}](URL) | {제목} | @본인 | → {KEY}.md |

**관련 정보**:
- **저장 경로**: `.docs/task/{브랜치명}/`
- **생성된 파일**: {N}개
- **Jira 에픽**: {Epic Key}
- **Jira 스프린트**: {Sprint Name}

#### 1) 지라 프로젝트 및 에픽 확인

**1. 프로젝트 키 확인**
사용자에게 작업을 생성할 Jira 프로젝트 키(예: `FE`, `PROJ` 등)를 확인한다.

**2. 기존 에픽(Epic) 조회 및 선택 (Optional)**
프로젝트 키가 확인되면, `jiraSearch` (JQL: `project = {PROJECT_KEY} AND issuetype = Epic AND status != Done`)을 사용하여 해당 프로젝트의 **활성 에픽 목록**을 가져와 사용자에게 보여준다.

사용자에게 다음과 같이 선택지를 제공한다:
- **목록에서 선택**: 표시된 에픽 목록 중 하나를 선택
- **새 에픽 생성**: 기능명으로 새 에픽을 먼저 생성한 뒤 하위 Story 연결
- **에픽 키 직접 입력**: 목록에 없는 에픽 키를 직접 입력
- **에픽 없이 생성**: 에픽 연결 없이 Story만 생성

**3. 활성 스프린트(Sprint) 조회 및 선택 (Optional)**
에픽 결정 후, 해당 프로젝트의 보드에서 **현재 진행 중인 활성 스프린트 목록**을 가져와 보여준다. (보드 ID 확인 필요 시 검색)

사용자에게 다음과 같이 선택지를 제공한다:
- **목록에서 선택**: 표시된 활성 스프린트 중 하나를 선택
- **스프린트 지정 안 함**: 백로그(Backlog) 상태로 생성

#### 2) 티켓 생성 실행

각 작업에 대해 `jiraCreateIssue`를 호출한다.

**지라 티켓 생성 규칙**:
- **이슈 타입**: Story
- **담당자(Assignee)**: **본인(currentUser)**으로 지정한다.
- **독립된 스토리 생성**: 모든 작업은 서브태스크(Sub-task)를 사용하지 않고 각각 **독립된 스토리(Story)**로 생성한다.
- **에픽 링크**: 선택된 에픽 키가 있으면 모든 Story에 연결한다.
- **스프린트**: 선택된 스프린트 ID가 있으면 모든 Story에 지정한다.
- **사전 조건**: Atlassian MCP 연결 확인.
 (불가 시 마크다운만 저장 후 안내) → 프로젝트 키 확인 → 에픽 연결 여부 확인.

**`jiraCreateIssue` 호출** — 타입: Story, summary: 작업 제목, description:
```
{설명 1~2문장}

- 선행: {작업 제목 또는 "없음"}
- 후행: {작업 제목 또는 "없음"}
- API: {엔드포인트 또는 "없음"}
- UI 상태: {states 또는 "N/A"}
```

**생성 후** — 이슈 키 반영 및 파일 리네임을 스크립트에 위임:
```bash
# FE-XX.md → JIRA-KEY.md 리네임 + 파일 내 jira 필드 업데이트
python3 ${CLAUDE_PLUGIN_ROOT}/scripts/update_jira_keys.py {디렉토리경로} '{"FE-01":"PROJ-101",...}' --project {프로젝트키}
```

이후 사용자에게 결과 요약 보고.

## 4. 주의사항

- **FE 경계**: 클라이언트 유효성 검사·에러 UI·로딩은 FE. SSR 서버 로직·API 핸들러 비즈니스 로직은 BE.
- **원문에 없는 작업 추가 금지**. 암시된 것은 "추가 고려 사항"에 제안.
- **BE 작업 미추출** — `api:` 필드에 참조만.
- **기술 스택 중립** — 사용자 언급 시 용어 조정.
- **원문 불명확** → `note: [확인 필요: ...]`.
- **Jira 실패** → `jira: 생성 실패 - {사유}` 기록, 나머지 계속.
- **이 스킬은 "무엇을"만** — "어떻게"는 plan mode.

## 5. 추가 고려 사항 (선택적)

원문 미포함이지만 FE 구현 시 검토 필요한 항목. Jira 미생성. 원문이 충분하면 생략.
